## Detailed Interview Breakdowns (Top Companies)

### Google

**Interview Process (SDE-2 / L4-L5):**
1. **Recruiter Screen** (30 min) — Background, motivation, role fit
2. **Phone Screen** (45 min) — 1-2 medium/hard DSA problems on Google Doc
3. **Onsite (Virtual Loop)** — 4-5 rounds:
   - 2× Coding (45 min each) — Hard LeetCode-level, emphasis on optimization
   - 1× System Design (45 min) — Distributed systems at scale (billions of users)
   - 1× Googleyness & Leadership (45 min) — Behavioral, collaboration, ambiguity
   - (Sometimes) 1× Additional Coding or Design round

**What Google Tests:**
- **Coding**: Expect hard-level problems. Two approaches are usually expected (brute force → optimal). Follow-up questions are common ("Now solve it in O(1) space").
- **System Design**: Design for Google-scale (billions of QPS). Topics: search, maps, YouTube recommendations, Gmail notification system, Google Drive sync.
- **Googleyness**: How you handle ambiguity, disagree respectfully, support teammates. They look for intellectual humility and collaborative problem-solving.

**Top Topics:**
- Dynamic Programming (medium-hard)
- Graph algorithms (BFS/DFS, shortest path)
- Sliding Window and Two Pointers
- Trees and recursion
- String manipulation

**Tips:**
- Use Google Docs for phone screen (no autocomplete, no syntax highlighting — practice this)
- Always discuss complexity before coding
- Google values multiple approaches: mention brute force, then optimize
- System Design: focus on scalability and data modeling, not just architecture

### Amazon

**Interview Process (SDE-2 / L5):**
1. **Online Assessment** (90 min) — 2 coding problems + work simulation
2. **Phone Screen** (60 min) — 1-2 coding problems + LP questions
3. **Onsite (Virtual Loop)** — 4-5 rounds:
   - 2× Coding + LP (60 min each) — DSA problem + Leadership Principle deep dive
   - 1× System Design + LP (60 min) — HLD with LP questions woven in
   - 1× Bar Raiser (60 min) — Mixed: could be coding, design, or pure behavioral
   - 1× Hiring Manager (45-60 min) — Behavioral, team fit, career goals

**Amazon Leadership Principles (Must Memorize):**
1. **Customer Obsession** — Start with the customer and work backwards
2. **Ownership** — Think long-term, act on behalf of the entire company
3. **Invent and Simplify** — Find new ways to simplify
4. **Are Right, A Lot** — Good judgment and instincts
5. **Learn and Be Curious** — Never done learning
6. **Hire and Develop the Best** — Raise the performance bar
7. **Insist on the Highest Standards** — Continually raise the bar
8. **Think Big** — Bold direction inspires results
9. **Bias for Action** — Speed matters in business
10. **Frugality** — Accomplish more with less
11. **Earn Trust** — Listen attentively, speak candidly
12. **Dive Deep** — Stay connected to details
13. **Have Backbone; Disagree and Commit** — Challenge decisions respectfully
14. **Deliver Results** — Focus on key inputs and deliver with quality
15. **Strive to be Earth's Best Employer** — Work to create a safer, productive environment
16. **Success and Scale Bring Broad Responsibility** — Be better every day

**Key Amazon Tips:**
- **Every round has LP questions**. Prepare 2-3 STAR stories for each LP
- **Bar Raiser** is the most unpredictable round — this person is not from your team and has veto power
- System Design: Amazon loves topics like order processing, inventory management, recommendation systems, notification services
- Coding: Medium-level LeetCode, but must be done fast and clean (25 min for coding, 20 min for LP)

**Amazon Leadership Principles → STAR Story Mapping**

Prepare at least one strong STAR story per LP. Reuse stories across LPs where honest — but emphasize different angles.

| LP | Sample Behavioral Question | STAR Focus (Action + Result) | Story Theme Ideas |
|----|---------------------------|------------------------------|-------------------|
| **Customer Obsession** | Time you went above and beyond for a customer | User impact metrics, trade-off you made for customer | On-call fix for customer outage; feature driven by user research |
| **Ownership** | Something outside your job you owned end-to-end | Scope you took without being asked; long-term fix vs patch | Owned prod incident root cause; cleaned tech debt before launch |
| **Invent and Simplify** | Simplified a complex system or process | Before/after complexity; automation | Replaced manual deploy with pipeline; simplified API surface |
| **Are Right, A Lot** | Good judgment under ambiguity | Data you used; when you changed mind | Architecture decision validated by metrics |
| **Learn and Be Curious** | Learned a new tech to solve a problem | What you learned; how fast | Picked up K8s/Kafka for migration; course + production use |
| **Hire and Develop the Best** | Mentored someone or improved team bar | Person's growth; hiring decision | Mentee promotion; interview calibration; hiring bar raise |
| **Insist on Highest Standards** | Raised quality bar when team accepted mediocrity | Standard you enforced; pushback | Code review standards; test coverage gate; SLA improvement |
| **Think Big** | Bold idea you proposed | Scale of vision; incremental steps | Multi-region design; platform instead of one-off |
| **Bias for Action** | Decided with incomplete information | Risk managed; speed vs perfection | Ship MVP; rollback plan; 80% data decision |
| **Frugality** | Delivered more with less | Cost saved; creative resource use | Reduced cloud spend; reused existing service |
| **Earn Trust** | Built trust after mistake or conflict | Candor; listening; follow-through | Post-mortem ownership; transparent status updates |
| **Dive Deep** | Found root cause others missed | Details you investigated | Log/trace analysis; metrics drill-down |
| **Have Backbone; Disagree and Commit** | Disagreed with manager/peer then committed | Respectful dissent; support after decision | Architecture disagreement; supported launch anyway |
| **Deliver Results** | Hit hard goal despite obstacles | Numbers; timeline; blockers removed | Met deadline; recovered slipped project |
| **Earth's Best Employer** | Improved team safety, inclusion, or culture | Team health; sustainable pace | On-call rotation fix; psychological safety in reviews |
| **Success and Scale Bring Broad Responsibility** | Impact at scale; sustainability | Scale of impact; responsibility taken | Privacy at scale; cost of scale; community impact |

**STAR template**: [InterviewPlaybook.md](InterviewPlaybook.md) · Deep behavioral prep: [§14 Leadership](../../01_TechGuide/14_Leadership_Behavioral_SystemDesign.md)

**Top Topics:**
- Arrays, Strings, HashMaps
- Trees (BST, Binary Trees)
- Graphs (BFS/DFS)
- Dynamic Programming (medium)
- System Design: microservices, message queues, caching

### Meta (Facebook)

**Interview Process (E4/E5 — IC level):**
1. **Recruiter Screen** (30 min) — Background, motivation
2. **Phone Screen** (45 min) — 2 coding problems (1 easy + 1 medium, or 2 medium)
3. **Onsite (Virtual Loop)** — 4 rounds:
   - 2× Coding (45 min each) — 2 problems per round (speed matters)
   - 1× System Design (45 min) — Design a system (Facebook-scale)
   - 1× Behavioral (45 min) — Past projects, leadership, conflict resolution

**What Meta Tests:**
- **Coding**: Speed is critical at Meta. You are expected to solve 2 problems in 45 minutes. Practice solving medium problems in 15-20 minutes.
- **System Design**: Design for Facebook-scale. Topics: News Feed, Messenger, Instagram Stories, Live Video, Marketplace.
- **Behavioral**: "Move Fast" culture. They want to see initiative, ownership, and impact.

**Top Topics:**
- Arrays and Strings (most common)
- Trees and Graphs
- Dynamic Programming
- Sliding Window
- Binary Search

**Tips:**
- Speed is everything — practice timed coding (15 min per medium problem)
- Meta coding interviews use a shared IDE (CoderPad) — practice there
- For System Design, Meta cares deeply about data modeling and news feed ranking
- Behavioral: prepare stories about impact, moving fast, and handling ambiguity

### Microsoft

**Interview Process (SDE-2 / L62-L63):**
1. **Phone Screen** (45-60 min) — 1-2 coding problems
2. **Onsite (Virtual Loop)** — 4 rounds:
   - 2× Coding (45-60 min each) — Medium problems, clean code emphasis
   - 1× System Design (45-60 min) — Practical system design
   - 1× Behavioral / "As Appropriate" (45 min) — Hiring manager round

**What Microsoft Tests:**
- **Coding**: Clean, well-structured code with proper error handling. They value readability. Medium-level problems; hard problems are less common than at Google.
- **System Design**: More practical than Google. Topics: Teams chat, OneDrive sync, Azure services, Xbox matchmaking.
- **Behavioral**: Growth mindset, collaboration, learning from failure.

**Tips:**
- Microsoft values clean code more than speed — take time to write readable code
- Use Visual Studio or any IDE you prefer (no Google Doc stress)
- System Design rounds may involve Azure-specific architecture knowledge
- Less LP-heavy than Amazon but still values cultural alignment

### Flipkart

**Interview Process (SDE-2 / SDE-3):**
1. **Online Assessment** (60-90 min) — 2-3 coding problems
2. **Phone Screen** (45-60 min) — 1-2 coding problems
3. **Onsite** — 3-4 rounds:
   - 1× Machine Coding (90-120 min) — Build a working system (LLD) from scratch
   - 1× Problem Solving (45-60 min) — DSA
   - 1× System Design (45-60 min) — HLD
   - 1× Hiring Manager (30-45 min) — Behavioral, team fit

**Machine Coding Round (Unique to Flipkart/Indian companies):**
This round requires you to build a working, extensible application in 90-120 minutes. You must demonstrate:
- Clean class design (SOLID principles)
- Design patterns (Strategy, Factory, Observer)
- Working code with proper input/output
- Extensibility for new requirements announced mid-round
- Unit-testable code

**Common Machine Coding Problems:**
- Parking Lot System

### Negotiation Strategy
1. **Never give a number first** — Let them make the offer
2. **Have competing offers** — The strongest negotiation leverage
3. **Negotiate on total comp, not just base** — RSU, sign-on bonus, relocation, WFH flexibility
4. **Ask for time** — "I appreciate the offer. Can I have a few days to consider?"
5. **Be specific** — "Based on my research and competing offer from X, I was expecting total comp closer to Y"
6. **Don't bluff** — Only mention competing offers if you actually have them
